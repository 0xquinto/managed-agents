## Findings n=10 surfaced that n=1 hid

### Triage `triage_payload.candidates` only passes 8/10

| Assertion | k/n | Wilson 95% CI |
|---|---|---|
| `envelope.triage_payload.candidates` (`min_length: 1`) | 8/10 | 0.800 [0.490, 0.943] |

The behavior is binary across 10 trials: 8 trials emit a 2-element candidates list with both registry contracts (`INS-2026-007`, `INS-2026-008`) and low scores (~0.15-0.30) + reasoning; 2 trials (`canonical_001`, `canonical_006`) emit `[]`.

Sample populated list (canonical_000):
```
[
  {"contract_id": "INS-2026-007", "score": 0.15, "reason": "Subject mentions Q1 analysis, matching timeline and content type of existing Tafi contract, but sender domain and address do not match registered Tafi addresses"},
  {"contract_id": "INS-2026-008", "score": 0.15, "reason": "Subject mentions Q1 analysis, but sender domain and address do not match registered Tucumán Capital addresses"}
]
```

The v2 prompt says "ranked `candidates` list of contract IDs with `score` ∈ [0,1] and `reason`" but doesn't mandate exhaustiveness. *Hypothesis (not directly observed — chain-of-thought isn't captured):* when Haiku assesses every registry row as low-confidence, it occasionally collapses to `[]` instead of reporting them as low-score candidates. Tightening the rule to require exhaustive listing on triage is a v3 candidate, but we should also test whether the empty-list rate is even *prompt-actionable* before assuming it is — the failure could equally be Haiku temperature variance regardless of prompt wording.

### Implications

- **v3 prompt iteration target:** require the candidates list to include every open registry contract on triage, even if all scores are low. Wording sketch: "On triage, `candidates` MUST list every registry row as a scored candidate; an empty list is never valid for triage and means the human reviewer has no signal to work with."
- **Production handling:** the poller's `ResolverStep` should treat an empty `candidates` on triage as a soft-fail (post a Teams card asking the human to look at this email manually) rather than crashing or auto-routing.
- **Eval-side note:** the `min_length: 1` threshold is correct — empty candidates is a real failure mode worth flagging, not a measurement artifact.

## Cost + cadence

- 40 trials × ~$0.01-0.05 each = roughly $0.40-2.00 spend.
- Wall-clock: ~50 minutes serial. Could parallelize at the slice level if we add a `--concurrent` flag to runner.py.
- For the production-grade A/B (paired-McNemar between v2 and a future v3), we'd want each variant at n=25 per slice = 200 trials per A/B pair = ~$5-10 + ~4 hours. That's the spend tier we'd reach for a "should we ship v3?" decision, not for hypothesis-testing iterations.
