## `.claude/agents/events-expert.md`

**Status:** `drift detected`

### Finding: `is_error` field missing from `user.custom_tool_result` event

The upstream API schema for `POST /v1/sessions/{session_id}/events` documents an `is_error` optional
boolean on the `user.custom_tool_result` event (both request params and returned event object). The
local file does not mention this field.

**Upstream excerpt** (source: `https://platform.claude.com/docs/en/api/beta/sessions/events/send`,
fetched via `mcp__Exa__web_fetch_exa`):

```
BetaManagedAgentsUserCustomToolResultEventParams = {
  custom_tool_use_id: string   // id of the agent.custom_tool_use event
  type: "user.custom_tool_result"
  content: optional array of BetaManagedAgentsTextBlock | BetaManagedAgentsImageBlock | BetaManagedAgentsDocumentBlock
  is_error: optional boolean   // Whether the tool execution resulted in an error.
}
```

And in the response object:
```
BetaManagedAgentsUserCustomToolResultEvent = {
  id: string
  custom_tool_use_id: string
  type: "user.custom_tool_result"
  content: optional array of ...
  is_error: optional boolean   // Whether the tool execution resulted in an error.
  processed_at: optional string
}
```

**Local lines needing update:**

- **Line 140 (example):**
  ```bash
  --event '{type: user.custom_tool_result, custom_tool_use_id: "EVENT_ID", content: [{type: text, text: "result"}]}'
  ```
  → Add `is_error` to the documented fields; example could show `is_error: true` for error cases

- **Lines 46–51 (user events table):** The `user.custom_tool_result` row describes it as
  "Response to custom tool call" with no mention of the `is_error` flag. Add a note that
  `is_error: true` signals an error result to the agent.

**Note:** `user.define_outcome` (local line 51) does not appear in the public API schema for the
send endpoint — this is expected since it is gated behind the `managed-agents-2026-04-01-research-preview`
beta header and the local file correctly annotates this requirement. Not drift.

Source: `https://platform.claude.com/docs/en/api/beta/sessions/events/send`

## Closing: Aggregate Recommendation

Three files need a follow-up edit pass (in priority order):

1. **`.claude/agents/agents-expert.md`** — Add `claude-opus-4-7` to the supported models
   table (lines 144–148) and update `--model` flag examples (lines 17, 40, 157). This is the
   highest-priority fix: `claude-opus-4-7` is the top-tier model in the upstream schema
   ("Frontier intelligence for long-running agents and coding") and the local file omits it
   entirely. Any new agent provisioned from this file will miss the best available model.

2. **`.claude/agents/events-expert.md`** — Add the `is_error: optional boolean` field to the
   `user.custom_tool_result` event documentation (line 140 example and lines 46–51 table).
   Without this, users have no visibility into signaling error conditions back to the agent.

3. **`.claude/agents/memory-expert.md`** — Remove the stale rule at line 312
   (`"CLI does not support memory-stores yet"`). The CLI surface is live upstream. The
   `## CLI Commands` block (lines 14–123) already documents the correct syntax; the contradictory
   rule risks confusing operators into using REST-only patterns when CLI is available.

Files to re-audit once upstream CLI pages deploy:
- **`files-expert.md`** — Re-check when `upload`, `delete`, `retrieve-metadata` pages go live.
- **`skills-expert.md`** — Re-check when `retrieve`, `delete`, `list`, and `versions:*` pages go live.
